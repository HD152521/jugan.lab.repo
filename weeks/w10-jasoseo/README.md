# w10 - 자소서 압축기

> 이 폴더는 **이번 주 목표·계획**. 실제 앱 코드는 `apps/jasoseo`(예정)에 둔다.
> 타깃: 취준생 · hub 실험 #10 · **시즌: 하반기 공채(9월)·상반기 공채(3월)에 맞춰 출시**

## 한 줄 컨셉
1,200자 자소서를 의미를 유지하며 정확히 500자로. 글자수와의 전쟁을 끝낸다.

## 관심 끄는 축 (주력 = 시즌 리텐션/실용)
- 🔁 **리텐션(주력): 회사별 버전 보관함** — 같은 문항의 500자/700자/1000자 버전을 저장·관리. 지원할 때마다 보관함 열러 옴. 공채 시즌마다 재유행.
- 🚀 바이럴(보조): "1200자 → 500자" before/after + 잘린 분량 시각화.

## 유입(모집) 전략
- 릴스 훅: 글자수 초과 빨간 표시(1,247/500) → 압축 → 딱 500 맞춤 (취준생이면 안 멈출 수 없는 화면)
- 시즌 타이밍: 공채 시즌 2주 전 출시, 대기업 공고 뜨는 날 맞춰 리포스트
- 저장 유도: "지원 시즌에 쓰려고 저장" 포지셔닝 — 저장 수가 알고리즘 신호

## 핵심 기능
- 글자수 목표 입력(500/700/1000/자유) → 압축 제안 (LLM API 사용 검토 — 이 실험은 규칙 기반으론 품질 한계)
- 압축 전/후 대비 뷰 + 어떤 문장이 빠졌는지 표시
- 공백 포함/제외 글자수 실시간 카운터 (이것만으로도 실용 가치)
- 회사·문항별 버전 보관함 (localStorage)

## 저장 방식
- 보관함은 로컬. LLM 사용 시 "입력 내용 저장 안 함" 명시 필수

## 완료 기준 (DoD)
- [ ] 대표 자소서 3편에서 의미 손실 없이 목표 글자수 ±5자
- [ ] 보관함 저장/불러오기/버전 관리
- [ ] 모바일 320px 깨짐 없음

## 리스크
- LLM 비용/키 관리 → 일 사용량 제한 or BYO 키 옵션 검토 (실험 중 유일한 API 비용 발생 지점)
- AI 자소서 논란 → "내가 쓴 글을 줄여주는" 도구로 포지셔닝 (생성 아님)
